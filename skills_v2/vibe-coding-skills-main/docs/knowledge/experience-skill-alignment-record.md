# 经验沉淀 Skill 对齐过程记录（案例驱动）

## 1. 文档目的

本记录用于沉淀一次真实的多轮对齐过程，作为后续新 Skill 设计与评审的背景知识。

本次案例的核心价值不在某个字段命名，而在于明确：

- Skill 的定位边界是什么
- 什么应该放进 Skill，什么不应该
- 如何把“多轮讨论才对齐”的经验转成可复用规则

## 2. 案例背景（为何需要该 Skill）

在实际讨论中，围绕以下问题发生了多轮反复对齐：

- `native_only / hybrid / mas_only` 是否是合理长期建模
- `alias_map` 是否存在语义冗余与行为重叠
- `fallback` 是预定义模式还是运行时重编排动作
- `run_audio_agent / video_generate_audio` 这类控制字段是否弱化了 orchestrator
- `smoke / release` 的概念边界是否被混淆

结论：该类问题具有跨项目复发性，且讨论成本高，适合沉淀为经验复用 Skill。

## 3. 对齐过程中暴露的典型问题

1. 术语与语义未对齐  
模式名、设计承诺、运行行为不一致，导致理解偏差和评审误判。

2. 目标与实现混淆  
例如“低 FP”是质量目标，但容易被误转译为“必须实现评分引擎”。

3. Skill 与系统功能边界模糊  
Skill 是先验指导/流程引导，不应被误设计为运行时策略系统。

4. 抽象层级混乱  
将“项目状态记忆”和“跨项目经验复用”混在同一语义层，造成重叠。

## 4. 最终共识（本次对齐结果）

1. Skill 的本质  
Skill 是可复用的先验知识与流程引导，可包含必要脚本，但不是运行时模块。

2. 新 Skill 的定位  
该 Skill 用于“经验提炼与复用”，不是日志归档，也不是 orchestrator 策略引擎。

3. 触发策略  
用户主动触发优先；可有“建议保存”提示，但必须低打扰、低 FP、需用户确认。

4. 低 FP 的实现方式  
通过硬条件与交互约束控制，不通过评分计算引擎实现。

5. 与 layered-project-memory 的关系  
二者正交：  
- layered-project-memory 负责项目连续状态  
- 经验沉淀 Skill 负责跨项目可复用规则  
允许通过引用关联，不做内容双写。

## 5. 新 Skill 的设计边界（必须遵守）

### 5.1 应该做

- 提供经验提炼模板（问题签名、反模式、决策规则、检查项、证据）
- 提供触发与反触发规则（何时问、何时不问）
- 提供确认后写入的最小流程
- 提供经验复用检索入口（用于设计/review场景）

### 5.2 不应该做

- 不实现运行时评分引擎
- 不替代 orchestrator 的执行决策
- 不接管项目状态机
- 不做高频自动写入

## 6. 建议的最小工作流（MVP）

1. `detect`  
识别是否满足经验沉淀硬条件（已收敛 + 可复用 + 有证据）。

2. `ask`  
只在满足条件时一次性询问：是否保存经验（是/否/稍后）。

3. `extract`  
按固定模板提炼经验卡。

4. `persist`  
确认后落库；建立 `source_event_refs` 或相关证据引用。

5. `reuse`  
后续类似问题通过问题签名与检查项快速检索。

## 7. 经验卡建议 schema（最小）

- `title`
- `problem_signature`
- `anti_pattern`
- `decision_rules`
- `review_checklist`
- `scope`
- `evidence_refs`
- `source_event_refs`
- `created_at`
- `updated_at`

## 8. 触发规则（低 FP 版本）

### 8.1 允许建议保存（全部满足）

- 问题已收敛（有明确结论）
- 经验可跨场景复用（非一次性细节）
- 至少有一条证据引用（代码/文档/测试/讨论结论）

### 8.2 禁止建议保存（任一命中）

- 仍在开放讨论，结论未定
- 仅是微小修复或临时 workaround
- 用户刚拒绝同类建议（冷却期内）

### 8.3 交互约束

- 同一主题仅提示一次
- 必须用户确认才写入
- 拒绝后不追问

## 9. 与 layered-project-memory 的协作建议

1. layered-project-memory 记录“这次问题如何演进”
2. 经验沉淀 Skill 记录“以后同类问题如何处理”
3. 双向引用但不复制正文，避免冗余与漂移

## 10. 后续行动建议

1. 先按本记录落地新 Skill 的 `SKILL.md` 与 references（中文）
2. 用本案例生成第一条经验卡作为基准样例
3. 在 Skill 完成后加入 Codex skills，通过主动触发进行经验沉淀
4. 后续评审以本记录为约束基线，防止再次概念漂移
